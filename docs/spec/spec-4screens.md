## 全体構造

```
現場  ／  仕舞い
```

現場：現場に向かう、入る、作業する、返す  
仕舞い：家・事務所など落ち着いた場所で記録・請求を整える

「迷ったら」「返す」はタブではなく、現場画面内の大きなボタンとして配置する。

## 画面遷移（時間軸）

```
画面1　現場トップ
  └─ 現場をタップ
       ↓
画面2　申し送り
  └─ 入館・作業開始
       ↓
画面3　作業中
  ├─ 迷ったら押す  → 画面4　SOS → 送信完了
  ├─ 撮影・メモ    → カメラ起動
  └─ 今日を返す   → 画面5　山返し → 完了
```

## 画面1 ── 現場トップ

### 役割
「今日はどこへ向かいますか？」から始まる起点画面。  
Hunterの最初の行動は、現場を登録することではなく、今日向かう現場を選ぶことである。

### 表示例
```
SENLINGS

今日はどこへ向かいますか？

［ 三井アウトレットパーク福岡 ］
福岡県　前回入場：2024-12-19

［ 20260417_福岡テスト現場 ］
福岡県
```

画面下部に小さく：
```
現場が見つからない場合
  → 担当者へ確認する
  → 現場を追加する
```

### 設計方針
- 「＋ 現場を追加」は前面に出さない
- 現場登録は原則として管理者・親方側の行為
- 一人親方・個人利用を考慮し、奥に導線は残す

| 1 | 現場アクセス | 住所・入口・受付・防災センター・GoogleMapで開く |
| 2 | 特殊装備・入場条件 | ヘルメット・安全帯・安全靴・入館証・火気使用届など |
| 3 | 駐車場 | 駐車可否・場所・高さ制限・満車時対応 |

## 画面4 ── SOS（迷ったら押す）

### 役割
問題を「上げる」画面。止まるかどうかはHunterが決める。

### 設計方針
**表は現場語。裏は管理語。AIが翻訳する。**

### 表側（Hunterが見る）
```
困ったことを選んでください。

［ 入れない・搬入できない    ］
［ 図面と違う              ］
［ 納まりがわからない       ］
［ 誰に聞けばいいかわからない ］
［ 危ない                 ］

［ 写真を追加 ］
［ ひとこと  ］

［ 送る ］
```

- 1回につき1つ選ぶ（複数カテゴリ同時展開しない）
- 写真・ひとことは任意
- 送信後、別の救助信号も送れる

### 裏側（AIが分類）

| 現場語 | 管理語 |
|-------|-------|
| 危ない | 安全 |
| 図面と違う | 品質 |
| 納まりがわからない | 品質 |
| 入れない・搬入できない | 工程 |
| 誰に聞けばいいかわからない | 工程 |
| 追加費用が出そう | 原価 |

### 送信後
```
伝わっています。

危ないときは、
待たずに電話してください。

［ 電話する ］
［ このまま待つ ］
［ 別の救助信号も送る ］
```

### 設計方針
- SOSは報告フォームではない
- 「止まれ」とは言わない。判断はHunterに残す
- 伝える導線と電話導線は必ず用意する

### データ
```json
{
  "sos_id": "sos_20241220_001",
  "site_id": "site_001",
  "timestamp": "2024-12-20T14:32:00",
  "display_text": "入れない・搬入できない",
  "category": "koutei",
  "note": "搬入遅延による",
  "photo": null,
  "status": "sent"
}
```

## 画面4-B ── SOS送信完了モーダル

### 役割
「送る」を押した後に表示される送信完了モーダル。
ただのシステム通知ではなく、裏側の精霊（Heart）が確かに受け取ったことを、静かに伝える。

### 構成

```
┌─────────────────────────┐
│ （画面4が暗転・80%黒）  │
│                         │
│    伝わっています。     │  ← Noto Serif JP・フェードイン
│                         │
│  止まるかどうかは、     │
│  あなたが決める。        │
│  伝わったことだけ、       │
│  お知らせしました。       │
│                         │
│  ［ 電話する ］          │  ← 警告オレンジ枠（「危ない」選択時のみ表示）
│  ［ このまま待つ ］      │
│  ［ 別の救助信号も送る ］│  ← テキストリンク
│                         │
└─────────────────────────┘
```

### 表示ルール
- 「電話する」ボタンは「危ない」選択時のみ表示
- 「このまま待つ」を押すと作業中画面（画面3）に戻る
- 「別の救助信号も送る」を押すと画面4のSOS選択に戻る

### 設計方針
- alertを廃止。世界観に沿ったモーダル画面で表現
- メッセージは Noto Serif JP（明朝体）でフェードイン
- システム語ではなく、血の通った「返答」として表現
- Brand Doctrine「HUBは灯火」を体現する画面

## 画面5 ── 山返し

### 役割
今日を返す画面。記録ではなく、痕跡を次のHunterへ渡す行為。

### 構造
1画面スクロール。ステップ分割なし。

### ① また入りたいか（2択）
```
［ また入りたい  ］
［ 複雑な気持ち ］
```
内部データは3段階で保持：`"mode": "positive" | "conditional" | "avoid"`

### ② 次に伝えたいこと（4カテゴリ、各任意）

| 表示ラベル | 管理語 |
|-----------|-------|
| 危なかったこと | 安全 |
| 仕上がり・納まり | 品質 |
| 段取り・待ち時間 | 工程 |
| 追加・材料・経費 | 原価 |

AIキャラクターの出現はここのカウントに連動（別仕様書参照）

### ③ 次の人へ一言（任意・フリーテキスト）
例：`搬入口は右側が使いやすい`

### ④ `今日を、返す。`

### 完了後
```
🔥

次のHunterへ、届きます。

今日の現場は、終わりました。
```

### 設計方針
- 「ONモード」「YORAIモード」はUI上に出さない
- 内部データ・キャラクター演出の裏側でのみ使用
- 世界観語は裏側からにじませる

### データ

  "site_id": "site_001",
  "timestamp": "2024-12-20T18:45:00",

    "anzen": { "selected": "特に問題なし", "note": "" },
    "hinshitsu": { "selected": "仕上がり良好", "note": "" },
    "koutei": { "selected": "少し遅れた", "note": "搬入遅延" },
    "genka": { "selected": "経費なし", "note": "" }

## 画面5 補足 ── 仕舞いタブ確認モーダル

### 役割
作業中・休憩中・SOS中・山返し中に仕舞いタブを押した際、
山返しを経ずに仕舞いへ移動しようとしていることを確認する。

### 表示条件
以下の画面表示中に仕舞いタブが押されたとき：
- 画面3（作業中）
- 画面3-B（休憩）
- 画面4（SOS）
- 画面5（山返し）

### 構成

```
┌─────────────────────────┐
│                         │
│  まだ「山返し」が       │  ← Noto Serif JP
│  終わっていません。      │
│                         │
│  このまま仕舞いへ向かうと、│  ← Ash Grey
│  今日の痕跡は残りません。  │
│                         │
│  ［ 今日を、返す。 ］    │  ← Ember Gold主導線（画面5へ）
│  ［ 痕跡を残さず仕舞う ］ │  ← 副導線（画面6へ・記録なし）
│  ［ 現場に戻る ］        │  ← キャンセル（テキストリンク）
│                         │
└─────────────────────────┘
```

### 文言の方針
- システムエラーではなく、静かに確認するトーン
- 「今日を、返す。」は画面5の下部ボタンと同じテキスト
  → 「あの儀式をやっていなかった」と直感的に気づかせる
- 「記録なし」ではなく「痕跡を残さない」という現場語に寄せる
- 「戻る」ではなく「現場に戻る」で接続先を意識させる

### 設計方針
- 背景は完全不透明（#0d0d0b）でスクロール時も透けない
- モーダル表示中は body のスクロールを止める
- 3つの選択肢は明確に分離

### 下部操作パネルの原則

画面下部のアクションボタンエリアとタブバーは「一体化した操作パネル（コックピット）」として設計する。

```css
/* アクションボタンエリア共通 */
.action-area {
  padding: 20px 20px 24px;
  background-color: #0d0d0b;  /* ベタ塗り（スクロール時の透過防止） */
  border-top: 1px solid #2a2a24;
  flex-shrink: 0;
}
/* タブバーとの二重線を避けるため、border-bottom は引かない */
```

- スクロール時にコンテンツが透けて見えないよう背景色をベタ塗りする
- タブバーとボタンエリアを視覚的に一体化させる
- この原則は画面3〜5のアクションボタンにも適用する

## 画面6 ── 仕舞い（帰り火）

### 役割
現場では使わない。家に帰ってから、今日を整える画面。

### 表示要素
- 今日の記録（現場名・作業時間・SOS件数・経費・山返しタグ）
- 今月の記録（カレンダー・稼働日数・時間合計・経費合計）
- 請求書下書き（AI生成 → Hunter確認 → PDF → 送信）
- **データをエクスポートする**（後述）
- 設定（名前・通知先メール・管理者モード）

### データエクスポート（必須機能）

```
仕舞い画面 → 「データをエクスポートする」

  ├─ JSON（全データ）
  │    workSessions・expenses・photos・helpSignals
  │    ※ アプリをやめるときに使う
  │
  ├─ CSV（月次 / 工事単位）
  │    freee・マネーフォワードへのインポート用
  │
  └─ Excel（請求書）
       藤田建装専用書式・将来複数テンプレート対応
```

**設計方針**
- エクスポートはHunterの権利。課金・制限をかけない
- 「いつでも持ち出せる」をUIで見える形にする
- 説明文：`このデータはあなたのものです。いつでも持ち出せます。`

### 請求書の流れ
```
workLogs・expenses の蓄積
  ↓
月末にAIが下書き生成
  ↓
Hunterが確認
  ↓
PDF確認 → メールで送信
```

### 設計方針
- 請求書の確定・送信は必ずHunterが行う（自動送信しない）
- AIは下書きを作る。判断と送信は人間が行う

## 用語浸透設計（20日間）

### 基本方針
- ボタン名は最後まで標準語で固定
- 補足文・完了文でSenlings語を染み込ませる
- 初日から世界観語を多用しない

### 用語の階段

| 段階 | 標準語 | 現場語 | Senlings語 |
|-----|-------|-------|-----------|
| 人 | 現場に入る人 | ものをつくる人 | Hunter |
| 場所 | 現場 | 今日入る山 | 山 |
| 事前確認 | 申し送り | 現場の作法 | 山の作法 |
| 困りごと | 迷ったら送る | 救助信号を送る | Signal |
| 記録 | 写真・メモ | 痕跡を残す | Trace |
| 終了 | 今日を返す | 次へ返す | 山返し |
| 帰宅後 | 今日の記録 | 今日を整える | 仕舞い |
| 完了感 | 次の人へ届く | 次へ巡る | ENが巡らせる |

### 3段階・20日間スケジュール

**第1段階（1〜5日目）：パターンA — 標準語**
- 世界観語をほぼ出さない
- 使い方がわかる・怖くないを優先

ボタン固定：
```
申し送りを見る  /  迷ったら送る  /  撮る・残す  /  今日を返す
```

完了文：`次の人へ、届きます。`

---

**第2段階（6〜15日目）：パターンB — 現場語を混ぜる**
- 補足文に現場語を追加
- 「これは管理アプリではない」と少しずつ伝える

追加する補足文：
```
困ったことは、現場語のまま送れます。
次に入る人が迷わないように、今日の痕跡を少しだけ残してください。
```

完了文：`次の人へ、届きます。今日の痕跡が残りました。`

---

**第3段階（16〜20日目以降）：パターンC — Senlings語を少し混ぜる**
- Hunter／山返しを補足文・完了文に入れる
- ボタン名は変えない

追加する補足文：
```
山返しです。次のHunterが迷わないように。
救助信号として届けます。
```

完了文：`次のHunterへ、届きます。今日の山返しが終わりました。`

---

### やってはいけないこと
- 初日から世界観語を多用する（山・救助信号・帰り火など）
- ボタン名を頻繁に変える
- AIキャラクター（HOO / KERO / KON / EN）を早く出しすぎる

### 1か月後の理想状態
```
申し送りを見る → 迷ったら送る → 今日を返す → 仕舞いで整える
```
この流れが体に入っている。
「次のHunterへ、届きます。」を見ても違和感がない。

---

## 設計原則

管理しない。判断を奪わない。

Senlingsは、Hunterを管理するためのアプリではない。  
現場に入る人へ必要な情報を届け、迷ったときに伝える導線を作り、今日の痕跡を次へ返すための道具である。

---

*現場では、押す・撮る・返すだけ。*  
*家では、今日を整える。*

*表は現場語。裏は管理語。AIが翻訳する。*

*知らなかっただけで、誰も怒鳴らなくていい。*
