# PHOTO_LOG_SPEC_v0
## Senlings v0 写真ログ仕様（現場内モード）

## 0. 目的
写真を「報告」ではなく「攻略ログ」にする。
写真は証拠であり、記録であり、未来への橋渡し。

---

## 1. Project ID と 命名規則（最重要・完全固定）
- **project_id の定義**: `{工事コード}_{短縮現場名}_{工期開始日YYYYMMDD}`
  - 例: `A2413_MINATOMIRAI_20260215`
  - ※ 工期開始日は登録後変更不可。延期・追加は別プロジェクトとする。
  - ※ 短縮現場名（slug）は英数字・ハイフンのみ（最大20文字）。
- **写真ファイル名**: `{project_id}_{HHMMSS}.jpg`
  - 例: `A2413_MINATOMIRAI_20260215_081423.jpg`
  - 人間が読め、CLIでの検索 (`ls *MINATOMIRAI*`) や時系列ソートが完璧に機能する。

---

## 2. v0の原則
1. 現場到着後は即カメラ起動
2. 文字入力を強制しない（保存最優先）
3. 写真は必ず project_id と紐付ける
4. 現場外（移動）と現場内（カメラ）のモードは絶対に混在させない

---

## 3. 撮影フローと UI原則
- `SiteModeScreen` → `CameraButton` → 撮影 → 自動保存 → `SiteMode`へ戻る
- 現場内ではボタンは大きく、余計なメニューは排除。保存成功の明確なフィードバックのみ。

---

## 4. 写真の分類（任意）
- 任意タグ: `before`, `after`, `issue`, `progress`
- 未選択でも即保存可能。分類は後回しでよい。

---

## 5. 保存と削除のルール
- **オフライン前提**: 端末ローカル保存必須（ネット接続不要）。
- **削除**: 物理削除可能だが、`event_log` に `photo_deleted` を記録（隠滅を防ぐ）。

---

## 6. けじめ（思想）
写真は「空気」を閉じ込める。現場は二度と同じ状態にならない。
Senlingsは、忘れない、責めない、ただ残す。
